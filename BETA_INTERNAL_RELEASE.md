# Внутренний beta-релиз DevPulse Android

## Версия beta

- Версия приложения: `1.19.0`
- Тип сборки для внутреннего тестирования: `staging`
- APK артефакт: `app/build/outputs/apk/staging/app-staging.apk`

## Provisioning Firebase для QA и разработки

- Получите `google-services.json` у владельца Firebase-проекта через закрытый канал доступа.
- Разместите файл в одном из путей:
  - `app/src/debug/google-services.json` для локальной разработки;
  - `app/src/staging/google-services.json` для QA/внутреннего beta.
- Убедитесь, что файл не попал в git (`.gitignore` уже включает стандартные пути).
- После provisioning выполните smoke-проверку запуска:
  - `./gradlew :app:testDebugUnitTest`
  - `./gradlew :app:assembleDebug`

## Release Notes (MVP beta)

### Что вошло в beta

- Авторизация и регистрация клиента.
- Стартовая маршрутизация на основе локальной сессии.
- Экран подписок: загрузка списка, добавление ссылки, удаление с подтверждением.
- Экран ленты обновлений с read/unread статусом и действием "пометить как прочитанное".
- Интеграция push (FCM): обработка payload, сохранение истории в Room, системные уведомления.
- Runtime permission для уведомлений (Android 13+) и экран настроек разрешений.
- CI quality gates для PR: `qualityCheck`, `testDebugUnitTest`, contract-выборка через `--tests`, `assembleDebug` + публикация отчетов и CI summary.

### Известные ограничения beta

- UI smoke-тесты (instrumented/Compose) запускаются автоматически в CI job `Instrumented Smoke (Compose)` с retry и артефактами диагностики.
- В `lintDebug` есть предупреждения, которые временно зафиксированы baseline (`app/lint-baseline.xml`).
- Текущие `BASE_URL` для `staging`/`release` остаются заглушками до подключения реального backend-контура.
- Матрица локальных/CI quality checks: `docs/ci-quality-matrix.md`.
- Матрица instrumented smoke-сценариев: `docs/instrumented-smoke-matrix.md`.
- Flaky/quarantine policy для smoke: `docs/instrumented-smoke-flaky-policy.md`.
- Архитектура хранения сессии: `docs/session-storage-architecture.md`.

## Smoke Checklist (реальное устройство)

Статусы для ручного прогона:

- `[ ]` PASS
- `[ ]` FAIL
- `[ ]` N/A

### 1. Установка и cold start

- `[ ]` Установить `app-staging.apk` на устройство.
- `[ ]` Первый запуск без краша.
- `[ ]` Приложение корректно открывает стартовый экран (Auth или Subscriptions по состоянию сессии).

### 2. Авторизация и сессия

- `[ ]` Успешный вход с валидным логином.
- `[ ]` Ошибки валидации и сетевые ошибки показываются пользователю.
- `[ ]` После перезапуска приложения активная сессия сохраняется.
- `[ ]` Logout очищает сессию и возвращает на экран авторизации.

### 3. Подписки

- `[ ]` Список подписок загружается корректно.
- `[ ]` Добавление валидной ссылки обновляет список без ручного перезапуска экрана.
- `[ ]` Невалидный URL блокируется валидацией формы.
- `[ ]` Удаление ссылки через confirm dialog работает корректно.

### 4. Push и уведомления

- `[ ]` Получение push с валидным payload создает запись в ленте обновлений.
- `[ ]` Для Android 13+ запрашивается `POST_NOTIFICATIONS`, после выдачи разрешения push виден в шторке.
- `[ ]` Для Android <13 push виден в шторке без runtime-разрешения.
- `[ ]` Системное уведомление открывает экран обновлений.
- `[ ]` Канал уведомлений `devpulse_updates` присутствует в системных настройках приложения.
- `[ ]` Payload без `content/body/description` сохраняется с fallback-сообщением и не вызывает crash.
- `[ ]` Payload без `url/link` или с невалидным URL игнорируется без crash (есть диагностический лог).

### 5. Лента обновлений

- `[ ]` События отображаются в порядке времени (новые сверху).
- `[ ]` "Пометить как прочитанное" меняет статус и сохраняет состояние после обновления экрана.
- `[ ]` Empty state отображается при отсутствии событий.

### 6. Стабильность

- `[ ]` Нет критичных ANR/Crash в основных сценариях.
- `[ ]` Навигация между экранами (`Subscriptions`, `Updates`, `Settings`) стабильна.

## Критерии приемки beta

- Пройден smoke-прогон на минимум одном реальном Android-устройстве.
- Критичные дефекты (blocker/major) отсутствуют или документированы с workaround.
- Артефакт `staging` передан команде тестирования вместе с этим документом.
